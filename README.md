# Мониторинг и наблюдаемость ServiceRequest

Данный документ описывает стек наблюдаемости (Observability Stack), используемый в приложении **ServiceRequest**. Стек разделен по трем основным столпам: **Логи**, **Метрики** и **Трейсы** (распределенная трассировка).

В проекте используются два модуля: `notification-service` и `request-service`.

---

## 1. Логирование (Logging) — ELK Stack
*Логи — это текстовые записи о событиях, произошедших в сервисе (ошибки, старт приложения, запросы).*

| Компонент | Назначение |
| :--- | :--- |
| **Filebeat** | **Челночный бегун.** Установлен на каждом сервере с сервисами (`notification-service` и `request-service`). Он читает логи из файлов приложений и отправляет их дальше по цепочке. Гарантирует, что логи не потеряются при временных проблемах сети. |
| **Logstash** | **Обработчик.** Получает логи от Filebeat, парсит их (например, достает уровень ошибки, timestamp), обогащает дополнительными полями и передает в Elasticsearch. |
| **Elasticsearch** | **Хранилище и поиск.** Сохраняет все логи и индексирует их. Благодаря этому мы можем искать по `requestId` или тексту ошибки за последние 24 часа. |
| **Kibana** | **Визуализатор логов.** Веб-интерфейс для просмотра сырых логов из Elasticsearch. Позволяет строить графики по частоте ошибок в разрезе двух сервисов. |

---

## 2. Метрики (Metrics) — Prometheus & Grafana
*Метрики — это числовые показатели состояния системы (количество запросов в секунду, время ответа, загрузка CPU).*

| Компонент | Назначение |
| :--- | :--- |
| **Prometheus** | **Сборщик метрик.** Периодически опрашивает эндпоинты `/metrics` наших микросервисов (`notification-service` и `request-service`). Забирает оттуда данные (например, сколько запросов упало с ошибкой 500) и хранит их у себя как временные ряды. |
| **Grafana** | **Панель приборов (Дашборды).** Подключается к Prometheus и рисует красивые графики. Позволяет увидеть нагрузку на сервисы в реальном времени, динамику очередей и потребление ресурсов. |

---

## 3. Трассировка (Tracing) — Jaeger
*Трейсы помогают отследить путь конкретного запроса через все микросервисы.*

| Компонент | Назначение |
| :--- | :--- |
| **Jaeger** | **Следопыт.** Позволяет увидеть полную картину прохождения запроса. Например, когда пользователь создает заявку (`request-service`), сервис отправляет уведомление (`notification-service`). Jaeger покажет, сколько времени занял каждый вызов, где была задержка и какие ошибки возникли. <br/><br/>* **TraceId** — уникальный идентификатор всего путешествия запроса. <br/>* **Span** — одна операция внутри путешествия (например: "авторизация в request-service", "запрос в БД", "вызов notification-service по gRPC"). |
